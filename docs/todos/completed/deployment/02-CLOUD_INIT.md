# Cloud-init First Boot

Part of [[../DEPLOYMENT]].

## Problem

The Terraform instance ([[01-OVH_TERRAFORM]]) comes up as a bare Ubuntu 24.04 host. We need a single user-data script that takes it from "fresh VM" to "serving requests on :443" with no second SSH step. Idempotent where it can be (re-runnable for debugging), but the happy path is one-shot.

## Approach

`deploy/cloud-init.yaml.tftpl` is a cloud-config YAML template. Terraform renders it with `templatefile()`, embedding the `docker-compose.prod.yml` and `Caddyfile` contents via `file()` so they land on disk as part of `write_files`. Stages:

### 1. Base packages + Docker

```yaml
package_update: true
package_upgrade: true
packages:
  - ca-certificates
  - curl
  - ufw
runcmd:
  - install -m 0755 -d /etc/apt/keyrings
  - curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
  - chmod a+r /etc/apt/keyrings/docker.asc
  - echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu $(. /etc/os-release && echo $VERSION_CODENAME) stable" > /etc/apt/sources.list.d/docker.list
  - apt-get update
  - apt-get install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

### 2. Firewall

```yaml
  - ufw default deny incoming
  - ufw default allow outgoing
  - ufw allow 22/tcp
  - ufw allow 80/tcp
  - ufw allow 443/tcp
  - ufw --force enable
```

### 3. App directory, compose, Caddyfile, .env

`write_files` drops:

- `/opt/app/docker-compose.yml` — the prod compose override from [[03-GHCR_IMAGE]].
- `/opt/app/Caddyfile` — the reverse proxy config from [[04-TLS_AND_DNS]].
- `/opt/app/.env` — generated at render time:
  ```
  POSTGRES_USER=postgres
  POSTGRES_PASSWORD=${random_password_at_template_time}
  POSTGRES_DB=nextrade
  DATABASE_URL=postgres://postgres:${password}@db:5432/nextrade
  APP_IMAGE=${app_image}
  MIGRATE_IMAGE=${migrate_image}
  DOMAIN=${domain}
  ```
  `random_password` generated by Terraform (`resource "random_password"`) so it's stable across re-applies but unique per environment. Not rotated automatically.

### 4. Pull images + bootstrap

GHCR images are public ([[03-GHCR_IMAGE]]) so no `docker login` step is needed. The bootstrap is a single script dropped via `write_files` at `/opt/app/bootstrap.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
cd /opt/app
docker compose --profile oneoff pull       # include migrate + seed images
docker compose run --rm migrate
docker compose run --rm seed
docker compose up -d                        # profile-gated services stay off
```

Invoked from `runcmd`:

```yaml
  - bash /opt/app/bootstrap.sh
```

`migrate` and `seed` are one-shot services in the prod compose that use the migrate image ([[03-GHCR_IMAGE]]) with `command: npm run db:migrate` and `command: npm run db:seed` respectively. Both depend on `db` being healthy and sit behind `profiles: ["oneoff"]` so `up -d` doesn't start them.

Seeds run once on first boot only. The seed files truncate + reseed and are idempotent, so re-running them on a cloud-init redo is safe but wipes demo state — acceptable for a PoC.

## Notes

- **Cloud-init logs** live at `/var/log/cloud-init-output.log`. First debug step on a broken boot.
- **Caddy's LE cert request** blocks briefly on DNS propagation — if the A record isn't in place yet ([[04-TLS_AND_DNS]]), Caddy will retry. No action needed beyond waiting.
- **Re-running.** `cloud-init clean && cloud-init init` re-executes user-data locally, useful while iterating. Don't rely on it in production — destroy + re-apply is the clean path.
- **Secrets in user-data.** OVH stores `user_data` in plaintext against the project metadata, so the generated Postgres password is readable to anyone with project access. Acceptable for a demo with no real user data. No GHCR token lives here — images are public.

## What Was Done

`deploy/cloud-init.yaml.tftpl` ships the four planned stages: Docker CE install via the official apt repo, UFW (22/80/443 only), `write_files` for `/opt/app/docker-compose.yml`, `Caddyfile`, `.env`, and `bootstrap.sh`, and a `runcmd` invocation that pulls the `oneoff` profile, runs migrate + seed, then `up -d`. The `.env` is rendered with the Terraform-generated Postgres password, the operator's Anthropic key, the basic-auth pair, and the GHCR image refs. `make cloud-logs` tails `/var/log/cloud-init-output.log` over SSH for first-boot debugging.
